# Agent Implementation Brief: Human or AI Code Quiz

Build a Nuxt + Convex weekend project where users choose a programming language, judge whether 10 real-looking code snippets were written by a human or AI, receive playful non-revealing feedback after each answer, and get a shareable result page with score, aggregate stats, and provenance links.

This brief is intended for an implementation agent. Prefer small, working increments over large speculative abstractions.

## Non-Negotiables

- Use Nuxt for the app and Convex for app data.
- No login.
- One language per run.
- Quiz length is exactly 10 questions.
- User must answer every question; no skipping and no changing answers.
- Each run must contain between 4 and 6 human snippets and between 4 and 6 AI snippets.
- Correctness is hidden until the results page.
- Snark after answers must be neutral and must not correlate with correctness.
- Result pages are unlisted but shareable by short slug.
- Public shared results must not show full snippets by default.
- Human snippets must be sourced from permissively licensed pre-2021 open-source code.
- AI snippets must be generated from behavioral specs, not manually edited.
- Dataset provenance must be public in repo files.
- If time gets tight, cut languages before cutting provenance.

## Recommended Defaults

- Package manager: `pnpm`.
- Framework: current stable Nuxt.
- Data backend: Convex.
- Styling: Tailwind if already installed during setup; otherwise plain Nuxt CSS is acceptable.
- Syntax highlighting: Shiki.
- Dataset/pipeline scripts: TypeScript under `scripts/dataset/`.
- Initial implementation should use mock/dev dataset records that match the final schema.

## Product Behavior

### Landing

Route: `/`

Show:
- Project title placeholder, e.g. `Human or AI?`.
- Hook: `Can you tell human code from AI code?`
- Language picker.
- Start button.
- Link to `/methodology`.
- Small text: `10 real code snippets. No login.`

Launch languages:
- `js`
- `ts`
- `python`
- `rust`
- `cpp`

Only show languages that are enabled and have enough approved imported pairs. During mock development, enable whatever mock languages exist.

### Optional Demographics

Feature flag: `ENABLE_DEMOGRAPHIC_PROMPTS`

Show after language selection and before session creation.

Question 1: `Which best describes you?`
- `professional`
- `hobbyist_student`
- `dont_code_regularly`
- `prefer_not_to_say`

Question 2: `How often do you use AI coding tools?`
- `daily`
- `sometimes`
- `rarely_never`
- `prefer_not_to_say`

Both are optional. Provide a clear skip path. Store answers only at session level. Never show individual demographic answers on public result pages.

### Quiz

Route can be `/quiz/:sessionId` or equivalent Nuxt route.

Display:
- Language.
- Progress indicator like `ROUND 04 / 10`.
- Syntax-highlighted code panel with subtle line numbers.
- Two large buttons: `Human` and `AI`.
- Optional keyboard shortcuts: `H` and `A`.

On answer click:
- Immediately disable both buttons.
- Record response time from snippet display to click.
- Optimistically move into a short snark interstitial.
- Fire Convex answer mutation in the background.
- Show a random neutral snark message for about 600-900ms.
- Advance to next question.

Neutral snark examples:
- `hmmm, you sure?`
- `bold choice.`
- `that confidence is legally admissible.`
- `the vibes have been logged.`
- `a linter somewhere just sat upright.`
- `we'll circle back to that.`
- `interesting. suspiciously confident.`

Do not reveal correctness during the quiz.

### Results

Route: `/results/:resultSlug`

If session is incomplete, show a non-final state: `This quiz is not completed yet.` Include CTA to start a new round.

If complete, show:
- Score, e.g. `7/10`.
- Language.
- Completed time.
- Share CTA for X.
- Aggregate summary when enabled/available.
- Compact answer cards for the 10 questions.

Each answer card should show:
- Question number.
- User guess.
- Correct label.
- Correct/incorrect state.
- Crowd stat when enough answers exist, e.g. `% guessed AI`.
- Human source link or AI provenance link.
- License/model metadata.
- Optional report link if enabled.
- Deprecation note if the pair was later removed from active rotation.

Do not show all full snippets by default on public result pages.

X share text example:

```text
I scored 7/10 spotting Human vs AI code in JavaScript. Try it: <url> #HumanOrAI
```

Avoid claims like `AI detector`, `proof`, or `benchmark` in share copy.

### Methodology

Route: `/methodology`

Include concise sections:
- This is a game/informal experiment, not a scientific AI detector benchmark.
- Human snippets come from permissively licensed open-source repos at pre-2021 refs.
- AI snippets are generated by GPT-5.5 from structured behavioral specs.
- Both labels are normalized: comments stripped, top-level names anonymized, same formatter per language.
- Human review is accept/reject only.
- Limitations: selection bias, no peer review, not a detector.
- Link to public dataset files in repo.

## State And Resume Behavior

- Create sessions only after language selection and optional demographics step.
- Completed result pages remain shareable indefinitely.
- Incomplete sessions can be resumed within 5 minutes after reload/navigation using local storage.
- There is no hard active in-page timer. If a user leaves the tab open, they can continue.
- Server may clean up very old incomplete sessions later, e.g. after 24 hours.
- Store recent seen pair IDs in local storage per language to reduce repeats.

Local storage should track:
- `sessionId`
- `resultSlug`
- `language`
- current question index
- pending answers
- `startedAt`
- `lastActiveAt`
- recently seen pair IDs by language

Before showing final results, flush pending answer writes. If flushing fails, show a retry state.

## Convex Data Model

Use these logical tables. Exact field syntax can follow Convex conventions.

### `pairs`

Imported approved dataset records.

Fields:
- `pairId`: stable public ID, e.g. `js_0001`.
- `datasetVersion`: e.g. `v1`.
- `language`: `js | ts | python | rust | cpp`.
- `status`: `approved | deprecated`.
- `humanDisplayCode`.
- `aiDisplayCode`.
- `displayLanguage`.
- `lineCountHuman`.
- `lineCountAi`.
- `licenseSpdx`.
- `humanSourceRepo`.
- `humanSourceCommit`.
- `humanSourcePath`.
- `humanSourceStartLine`.
- `humanSourceEndLine`.
- `humanSourceUrl`.
- `aiModel`.
- `aiGeneratedAt`.
- `specModel`.
- `specGeneratedAt`.
- `specSummary` or full `spec` object.
- `aiProvenanceUrl` or dataset anchor if available.
- `createdAt`.
- `updatedAt`.

Indexes:
- by `language,status,datasetVersion`.
- by `pairId`.

### `sessions`

Fields:
- `resultSlug`.
- `language`.
- `datasetVersion`.
- `status`: `in_progress | completed | expired`.
- `profileCodingExperience`: optional.
- `profileAiToolUsage`: optional.
- `score`: optional until complete.
- `total`: should be 10.
- `startedAt`.
- `lastActiveAt`.
- `completedAt`: optional.
- `resumeExpiresAt`: startedAt + 5 minutes for reload resume.

Indexes:
- by `resultSlug`.
- by `status,language`.

### `sessionQuestions`

Fields:
- `sessionId`.
- `questionId`: stable per session question.
- `order`: 0-9.
- `pairId` or Convex pair doc id.
- `shownLabel`: `human | ai`.

Indexes:
- by `sessionId,order`.
- by `sessionId,questionId`.

### `answers`

Fields:
- `sessionId`.
- `questionId`.
- `guess`: `human | ai`.
- `shownLabel`: copied for immutable event simplicity.
- `isCorrect`.
- `responseMs`.
- `createdAt`.

Rules:
- `(sessionId, questionId)` is immutable/idempotent.
- Same duplicate guess returns success.
- Different duplicate guess is rejected; first answer wins.

Indexes:
- by `sessionId,questionId`.
- by `sessionId`.

### Aggregate Tables

Use separate aggregate tables or a single typed aggregate table. Keep it simple.

Required aggregate counters:
- Per language completed sessions.
- Per language total score.
- Per language score histogram buckets `0..10`.
- Per pair and shown label: shown count, human guesses, AI guesses, correct count, total response time.
- Optional demographic segment aggregates when enabled and answered.

Do not use incomplete sessions for score/percentiles. Submitted answers from incomplete sessions may count for per-question guess distributions.

## Convex Operations

Implement these logical operations.

### `startSession`

Input:
- `language`
- optional demographic answers
- `recentPairIds`: array from local storage

Behavior:
- Validate language is enabled.
- Filter approved pairs by language and dataset version.
- Exclude recent pair IDs if enough remain.
- Sample 10 distinct pairs server-side.
- Assign `shownLabel` so each class count is between 4 and 6.
- Create session and sessionQuestions.
- Create result slug at session start.
- Return client-safe quiz payload.

Payload includes:
- `sessionId`
- `resultSlug`
- `language`
- `questions`: `questionId`, `displayCode`, `displayLanguage`, `lineCount`
- `featureFlags`
- `resumeExpiresAt`

Payload excludes:
- correct label
- pair ID if avoidable
- source URL
- AI model/spec
- aggregate stats

### `submitAnswer`

Input:
- `sessionId`
- `questionId`
- `guess`
- `responseMs`

Behavior:
- Validate session exists and is in progress.
- Validate question belongs to session.
- Enforce idempotency.
- Compute `isCorrect = guess === shownLabel`.
- Store immutable answer.
- Increment per-question aggregate counters.
- Update session `lastActiveAt`.
- Return only success/idempotency status. Do not return correctness.

### `completeSession`

Input:
- `sessionId`

Behavior:
- Validate all 10 answers exist.
- Compute score from stored answers.
- Mark session completed.
- Store score and completedAt.
- Increment language score aggregates and optional demographic aggregates.
- Return `resultSlug`.

### `getResultBySlug`

Input:
- `resultSlug`

Behavior:
- If incomplete, return incomplete state.
- If complete, return compact result payload with score, answer cards, source/provenance metadata, and aggregates where enabled/available.
- Do not return all full snippets by default.

### `getEnabledLanguages`

Return enabled languages with approved pair counts and display names.

## Snippet Rules

- Prefer standalone functions for v1.
- Classes are deferred unless short and self-contained.
- Target display length: 15-45 lines.
- Hard cap: about 70 lines.
- Reject below about 8 meaningful lines unless unusually interesting.
- Strip comments and docstrings.
- Format Human and AI with the same formatter.
- Preserve local variable names.
- Anonymize top-level function/class names and project-specific identifiers.
- Keep type annotations/signatures as-is after anonymization.
- Allow obvious helper calls only if they do not hide the core logic.
- Hide repo/file/source until results.

Formatters:
- JS/TS: Prettier.
- Python: Black.
- Rust: rustfmt.
- C++: clang-format with fixed style.

## Human Source Policy

- Use curated `scripts/dataset/repos.yml`, not open-ended GitHub search.
- Clone repos locally at pinned pre-2021 refs.
- Source only permissive licenses for v1: MIT, Apache-2.0, BSD-2-Clause, BSD-3-Clause, ISC, Zlib, Boost.
- Exclude GPL, AGPL, LGPL, MPL, EPL for v1.
- Prefer credible but not meme-famous repos.
- Avoid instantly recognizable codebases/functions.
- Store exact commit SHA and source line anchors.
- Generate `dataset/v1/ATTRIBUTIONS.md`.

## AI Generation Policy

- UI asks `Human or AI?`.
- Dataset discloses AI snippets generated by GPT-5.5.
- Spec generation and code generation may use the same model family, but must be separate calls with separate prompt versions.
- Generate specs from normalized/anonymized human display code plus minimal metadata.
- AI implementation should match language, signature shape, input/output shape, sync/async nature, and type richness.
- Do not force same local variable names or exact algorithm.
- Automated retries are allowed only for parse/format failures.
- Retry up to 2 times with validation error context.
- Human review is accept/reject only.
- No manual editing of AI code.
- Publish approved pairs only.

## Acceptance Checks

The v1 app is acceptable when:

- A user can choose a language and start a 10-question quiz.
- The server chooses the quiz assignment, not the client.
- A run always has 4-6 human snippets and 4-6 AI snippets.
- The client never receives correct labels before results.
- Answers are immutable and idempotent.
- UI advances quickly after each answer while writes happen in the background.
- Results wait for pending answers to flush.
- Result slug can be opened in another browser and shows the completed result.
- Public result view does not show all full snippets by default.
- Aggregate stats update on answer submit/session completion.
- The methodology page exists.
- Feature flags can disable demographics and reporting.
- Mock dataset import works from JSONL.
- Dataset records validate against JSON Schema.

## Defer Explicitly

- Domain/name decision.
- Public leaderboard.
- Login/accounts.
- Confidence input.
- Showing full snippets on shared result pages.
- Complex admin UI for feature flags or review.
- Full executable tests for every snippet.
- GPL/copyleft source repos.
- Sophisticated bot/anti-cheat protections.
- Scientific claims about AI detection.
